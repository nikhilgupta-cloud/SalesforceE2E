🗺️ Overview

This framework combines traditional Playwright automation with a Claude AI backend to deliver a fully autonomous QA pipeline. Drop in a user story — the pipeline handles everything else: it reads the story, scrapes your live application for real DOM selectors, generates executable test code, runs the suite, and heals any failures automatically.

🔍 Live DOM Scraping

Playwright CDP scraper extracts real selectors from your live org — no more guessing label text. Results cached 24h.

🤖 AI Test Generation

Claude CLI reads user stories + domain knowledge and writes production-ready Playwright TypeScript. No hallucinated selectors.

🔧 Self-Healing

Failed tests are automatically analysed, classified, patched, and re-run — up to 3 rounds, with a healing report written to disk.

📊 Live Dashboard

Auto-refreshing HTML dashboard tracks every step of the pipeline and every test result in real time.

📎 Smart Change Detection

MD5 hashing means only changed user stories trigger regeneration — unchanged stories are never re-processed.

🚀 GitHub Auto-Push

On success, the pipeline commits generated tests and results with a timestamped message and pushes to your branch.

ℹ️
No ANTHROPIC_API_KEY required. The framework uses the Claude Code CLI (claude -p) which authenticates through your active Claude Code session. Ensure claude --version works in your terminal before running the pipeline.

⚙️ 8-Step Pipeline

The pipeline is orchestrated by scripts/run-pipeline.ts and tracked in real-time on the dashboard. Each step updates reports/pipeline-state.json as it runs.

🔍

Step 0 — Scrape Locators Cached 24h

Launches a headed Chromium browser with your saved session, navigates to each object's New-record form, and uses CDP DOM evaluation to extract all lightning-* component selectors — piercing Shadow DOMs. Results are written to knowledge/scraped-locators.json and injected into Claude's system prompt at Step 1. Skipped automatically if cache is less than 24 hours old.

🤖

Step 1 — AI Test Generation Smart delta

Scans user-stories/ and prompts/user-stories/ for .md files. Each user story is hashed (MD5). New stories generate fresh test blocks; changed stories regenerate only their block; unchanged stories are skipped entirely. Generated code is wrapped in // ── US-XXX START ── / END markers and appended to the relevant spec file.

📋

Step 2 — Verify Scenarios

Confirms that generated/test-scenarios/{object}-scenarios.md exists for each object in framework-config.json. Warns (but does not abort) if an object has no scenario file — meaning no user stories were provided for that object. Counts total scenario rows and reports to the dashboard.

📄

Step 3 — Verify Test Plan

Confirms that generated/test-plan.md exists. Dynamically counts the total number of test( calls across all spec files and user story sections across all scenario files, and updates the test plan counts in place.

🧪

Step 4 — Execute E2E Tests Headed Chromium

Spawns npx playwright test with all spec files in order (Account → Contact → Opportunity → Quote). Worker count is fixed at 1 — sequential execution is required for Salesforce session state management. The DashboardReporter custom reporter updates Step 4 status internally as tests run.

🔧

Step 5 — Self-Heal Failures Up to 3 rounds

Reads reports/results.json. For each failed test: extracts the test block, classifies the failure type (selector / timing / data / environment), asks Claude to generate a fix, applies the patch to the spec file, and re-runs that single test to verify. Repeats up to 3 rounds. Writes reports/healing-report.md with full diagnosis.

⚙️

Step 6 — Generate Final Scripts

Copies the post-healing spec files from tests/ into generated/scripts/. This gives you a clean snapshot of the final, production-ready test files at each pipeline run.

🚀

Step 7 — Push to GitHub Optional

Stages generated/, reports/, and creates a timestamped commit: chore: QA run YYYY-MM-DD — X/Y tests passing. Pushes to GITHUB_BRANCH. Probe scripts (probe-*.ts) are deleted before staging. auth/session.json and .env are never staged.

🤖 AI Agents

The framework uses 7 specialised Claude agents, each defined as a markdown file under .claude/Agents/. They connect in sequence — each one's output feeds the next.

1
Knowledge Base Agent
Loads the correct domain knowledge file from knowledge/agentforce-rm/ for the object under test. Prepends verified field names, UI patterns, and platform gotchas to every Claude prompt — preventing hallucination.
📁 .claude/Agents/agent-1-knowledge-base.md
2
Story Reader Agent
Parses user story markdown files. Handles both single-file format (prompts/user-stories/) and multi-story files (user-stories/). Detects object context from OBJECT N: NAME headers and extracts structured AC lists.
📁 .claude/Agents/agent-2-story-reader.md
3
Test Plan Drafter Agent
Generates generated/test-plan.md — a summary table of all objects, TC counts, user stories, test data strategy, execution order, risks, and entry/exit criteria.
📁 .claude/Agents/agent-3-test-plan-drafter.md
4
Scenario & Test Drafter Agent
The primary test-writing agent. For each AC: generates 1 positive + 1 negative + 1 edge case scenario, then writes the Playwright TypeScript test block. Uses scraped locators first, falls back to LWC filter patterns. Wraps output in // ── US-XXX START/END ── markers.
📁 .claude/Agents/agent-4-scenario-drafter.md
5
Test Executor Agent
Runs pre-flight checks (session.json, .env, stale .js files), then launches Playwright against all spec files in execution order (Account → Contact → Opportunity → Quote). Workers fixed at 1. Writes reports/results.json.
📁 .claude/Agents/agent-5-test-executor.md
6
Self-Healer Agent
Classifies each failure as selector / timing / data / environment. For selector and timing failures: re-probes DOM, cross-references scraped locators, patches the spec, re-runs. Up to 3 rounds per failure. Writes reports/healing-report.md.
📁 .claude/Agents/agent-6-self-healer.md
7
Dashboard & Reporter Agent
Aggregates all pipeline outputs into the live HTML dashboard, updates pipeline-state.json, generates the PowerPoint report via node scripts/generate-ppt.js, and optionally commits + pushes everything to GitHub.
📁 .claude/Agents/agent-7-dashboard-reporter.md

📖 Writing User Stories

The framework supports two story file formats. Both are scanned automatically by the pipeline.

Format A — Single-file per story (prompts/user-stories/)

Best for individual stories added one at a time. File name doesn't matter — what matters is the heading.

# US-001: As a Sales Rep, I want to create a Quote

**Object:** Quote

## Acceptance Criteria
AC-001-01: Given a valid Opportunity, when I click New Quote, a Quote record is created
AC-001-02: The Quote Status should default to "Draft"
AC-001-03: Required fields (Account, Opportunity, Expiry Date) must be validated

Format B — Multi-story file (user-stories/)

Best for exporting a full Jira sprint or project backlog. Group stories by object using OBJECT N: NAME headers.

OBJECT 1: QUOTE

## US-005 — As a Sales user, I want the system to mark the Quote as 'Accepted'
As a Sales Rep,
• I should be able to hand over the Quote to Legal by clicking 'Ready For Acceptance'...

OBJECT 2: OPPORTUNITY

## US-010 — As a Sales user, I want to convert an Opportunity to a Quote
...
⚠️
Object detection: In multi-story files, the parser tracks the current object from OBJECT N: NAME headers. All ## US-XXX stories below a header belong to that object until the next OBJECT header. Make sure your object names match one of the keys in framework-config.json.

Hash-based change detection

The pipeline stores an MD5 hash of each processed story in prompts/user-stories/.story-hashes.json. On subsequent runs:

Story stateAction taken
New story (no hash entry)Full generation — scenarios + spec block created
Changed story (hash mismatch)Old blocks removed, regenerated from scratch
Unchanged story (hash match)Skipped — nothing regenerated
💡
Force regeneration: Delete prompts/user-stories/.story-hashes.json to treat all stories as new and regenerate everything from scratch.
🚨
No hash store + existing marker blocks = duplicate tests. If .story-hashes.json does not exist (e.g. first run after cloning, or after deleting it), the pipeline treats every story as new — meaning it skips the "remove old block" step and goes straight to appending. If a // ── US-XXX START/END ── block already exists in a spec file, you will end up with two identical blocks for that story.

Fix before running: Manually remove any existing // ── US-XXX START ── to // ── US-XXX END ── blocks from the spec file, then run the pipeline — it will generate and append cleanly. Alternatively, clear the generated scenario file for that object: 
# Git Bash / macOS:
echo "# Test Scenarios — Quote" > generated/test-scenarios/quote-scenarios.md

# PowerShell (Windows):
Set-Content generated\test-scenarios\quote-scenarios.md "# Test Scenarios — Quote"

# CMD (Windows):
echo # Test Scenarios — Quote > generated\test-scenarios\quote-scenarios.md

🎫 Jira Integration

The framework can automatically pull user stories from Jira and convert them into the multi-story markdown format expected by generate-tests.ts — all without any manual copy-paste. This is handled by scripts/fetch-jira-stories.ts, which runs as Step -1 at the start of every pipeline when JIRA_BASE_URL is set.

ℹ️
Automatic: When JIRA_BASE_URL is present in your .env, the pipeline runs the Jira fetch before any other step — no extra command needed. If JIRA_BASE_URL is not set, Step -1 is silently skipped and the pipeline uses whatever story files already exist in user-stories/.

Setup — required .env variables

JIRA_BASE_URL=https://yourcompany.atlassian.net
JIRA_EMAIL=you@yourcompany.com
JIRA_API_TOKEN=your_personal_api_token      # NOT your password — generate from Atlassian account settings
JIRA_PROJECT_KEY=CPQ                        # The Jira project key to pull from

Optional variables

VariableDefaultPurpose
JIRA_JQL project = KEY AND issuetype = Task ORDER BY created ASC Override the JQL query — e.g. filter by sprint, epic, or label
JIRA_AC_FIELD — Custom field ID for Acceptance Criteria (e.g. customfield_10016). If set, ACs are read from this field instead of the description body.
JIRA_OBJECT_FIELD — Custom field ID that identifies the Salesforce object for a ticket (e.g. customfield_10020). Takes highest priority in object detection.
JIRA_OUTPUT_FILE user-stories/CPQ_User_stories.md Override the output file path for fetched stories

How to generate a Jira API token

  1. Go to Atlassian account settings Navigate to id.atlassian.com → Security → API tokens
  2. Click "Create API token" Give it a label like SalesforceE2E Framework so you can identify it later.
  3. Copy the token immediately It is shown only once. Paste it into .env as JIRA_API_TOKEN.

Object detection — how Jira tickets are mapped to Salesforce objects

The fetcher uses a four-tier priority system to decide which object a story belongs to. The first match wins:

PrioritySourceExample
1 — Custom field Value of JIRA_OBJECT_FIELD on the ticket Field value "Quote" → maps to quote object key
2 — Jira component Ticket's Component(s) field Component "quote" or "Quote CPQ" → quote
3 — Jira label Ticket labels matching sf-{key}, salesforce-{key}, or {key} Label sf-quote or quote → quote
4 — Keyword scan Word-boundary regex on ticket summary, then description Summary contains "quote" → quote
⚠️
Unresolved stories: If a ticket's object cannot be detected by any of the four methods, it is written to the output file as an HTML comment block labelled ⚠ UNRESOLVED STORIES. The pipeline will log a warning for each one. Fix by adding a Jira component matching an object key in framework-config.json, or a label like sf-quote.

How ACs are extracted

If JIRA_AC_FIELD is set, acceptance criteria are read from that custom field directly. Otherwise the fetcher looks for an "Acceptance Criteria" heading inside the ticket description and collects all bullet lines beneath it. Each AC is automatically numbered in AC-NNN-NN format if it doesn't already have one.

Output format

Stories are written to user-stories/CPQ_User_stories.md (or JIRA_OUTPUT_FILE) in the multi-story format, grouped by Salesforce object:

Salesforce CPQ — User Stories & Acceptance Criteria
Source: Jira CPQ — fetched 2026-04-09
Flow: Account → Contact → Opportunity → Quote (CPQ)
________________________________________
OBJECT 1: QUOTE

## US-005 — As a Sales user, I want the system to mark the Quote as 'Accepted'
As a Sales Rep...
Acceptance Criteria:
• AC-005-01: When I click 'Ready For Acceptance'...
• AC-005-02: System triggers all existing create order validations...
<!-- Jira: CPQ-123 -->

________________________________________

# Contents: Read and write
# Metadata: Read-only (required)

⚠️

If Step 7 fails

ErrorCauseFix
remote: Permission to … denied Token missing repo scope or wrong user Regenerate PAT with repo scope; ensure the token matches the repo owner
! [rejected] main → main (fetch first) Remote has commits not in local history Run git pull --rebase origin main then re-run the pipeline

🎯 Locator Strategy

Salesforce Lightning wraps form elements in Shadow DOMs. The framework uses a three-tier locator strategy — always prefer higher-priority patterns.

PriorityPatternWhen to use
1 — Scraped API Name [data-field-api-name="AccountId"] input Always first choice. Immune to UI label changes. Provided by knowledge/scraped-locators.json.
2 — ARIA Role getByRole('button', { name: 'Save', exact: true }) Buttons, links, tabs, dialog roles. Always use exact: true to avoid partial matches.
3 — Label-Based getByLabel('Account Name') Preferred over text-filter for form fields. Matches the <label> association directly.
4 — LWC Text Filter page.locator('lightning-input').filter({ hasText: /Account Name/i }).locator('input') Fallback when API name and label are not available. Use /regex/i for case-insensitive matching.
5 — Shadow Fallback page.locator('label:has-text("Name")').locator('xpath=..').locator('input') Last resort only. Used when LWC structure prevents all other patterns. Avoid if possible.

Key Playwright rules for Salesforce LWC

// ✅ DO — Always wait for visibility, never check with isVisible()
await page.locator('.my-selector').waitFor({ state: 'visible', timeout: 30000 });

// ❌ NEVER — networkidle hangs forever on Salesforce
await page.waitForLoadState('networkidle');  // banned

// ✅ Modal — always scope to non-error, visible dialogs
const modal = page.locator('[role="dialog"]:not([id="auraError"]):not([aria-hidden="true"])');

// ✅ After every page.goto() — dismiss Aura errors
await dismissAuraError(page);

// ✅ Combobox — click trigger, wait for option to attach
const trigger = page.locator('lightning-combobox').filter({ hasText: /Stage/i }).locator('button').first();
await trigger.click({ force: true });
const option = page.locator('[role="option"]').filter({ hasText: /Closed Won/i }).last();
await option.waitFor({ state: 'attached', timeout: 5000 });
await option.click({ force: true });

🧩 Patterns Reference

All 14 reusable interaction patterns for Salesforce Lightning (LWC). These are the canonical implementations — always prefer higher-priority locators (see Locator Strategy). Source of truth: .claude/Skills/automate/PATTERNS.md.

ℹ️
Decision hierarchy applies inside every pattern. For each field: try API name first → ARIA role → label → LWC text filter → shadow fallback. Never skip levels. Never invent selectors.

Pattern A — Text Input

Standard text fields rendered as lightning-input.

// Preferred: label-based
const field = page.getByLabel('Account Name');
await field.waitFor({ state: 'visible', timeout: 30000 });
await field.fill('Acme Corp');

// Fallback: LWC text filter
const field = page.locator('lightning-input').filter({ hasText: /Account Name/i }).locator('input');
await field.waitFor({ state: 'visible' });
await field.fill('Acme Corp');

Pattern B — Combobox (Picklist)

Picklist fields rendered as lightning-combobox.

const trigger = page.locator('lightning-combobox')
  .filter({ hasText: /Stage/i }).locator('button').first();
await trigger.click({ force: true });

const option = page.locator('[role="option"]').filter({ hasText: /Closed Won/i }).last();
await option.waitFor({ state: 'visible' });
await option.click();

Pattern C — Lookup (Record Search)

Relationship fields rendered as lightning-lookup.

const input = page.locator('lightning-lookup').filter({ hasText: /Account/i }).locator('input');
await input.fill('Acme');

const option = page.locator('[role="option"]').filter({ hasText: 'Acme Corp' }).first();
await option.waitFor({ state: 'visible' });
await option.click();

Pattern D — Checkbox

const checkbox = page.locator('lightning-input[type="checkbox"]').locator('input');
await checkbox.check();
await expect(checkbox).toBeChecked();

Pattern E — AG-Grid (RLM Line Editor)

Revenue Cloud line item grids use AG-Grid. Cells require click-then-enter to activate the inline input.

const cell = page.locator(`div[role="row"][row-id="row-0"] div[col-id="Quantity"]`);
await cell.click({ force: true });
await page.keyboard.press('Enter');

const input = page.locator('input.slds-input').last();
await input.fill('10');
await page.keyboard.press('Enter');
await waitForRlmSpinners(page); // always wait after grid edits

Pattern F — API Name Locator (MANDATORY if available)

When knowledge/scraped-locators.json has a data-field-api-name for a field, this MUST be used — it is immune to UI label renames.

const field = page.locator('[data-field-api-name="AccountId"] input');
await field.waitFor({ state: 'visible' });
await field.fill('Acme Corp');

Pattern G — Modal-Scoped Interactions

ALWAYS scope all interactions inside a dialog to the modal container. Never query globally when a modal is open.

const modal = page.locator('[role="dialog"]:not([id="auraError"]):not([aria-hidden="true"])');
await modal.waitFor({ state: 'visible', timeout: 30000 });

await modal.locator('lightning-input').filter({ hasText: /Name/i }).locator('input').fill('Test Value');
await modal.getByRole('button', { name: 'Save', exact: true }).click();
🚫
NEVER call page.locator(...) for fields when a modal is open — always use modal.locator(...)

Pattern H — Toast Validation

const toast = page.locator('.slds-notify_toast');
await toast.waitFor({ state: 'visible', timeout: 10000 });
await expect(toast).toContainText('was created');
await toast.waitFor({ state: 'hidden', timeout: 15000 }); // wait to dismiss before chaining

Pattern I — iFrame Handling

const frame = page.frameLocator('iframe[title="Your Frame Title"]');
await frame.locator('button:has-text("Save")').click();

Pattern J — Safe Click (Retry Wrapper)

Use sparingly — prefer fixing the root cause (missing waitFor) over wrapping in retry.

async function safeClick(locator: Locator): Promise<void> {
  for (let i = 0; i < 3; i++) {
    try {
      await locator.click();
      return;
    } catch {
      await locator.page().waitForTimeout(500);
    }
  }
  throw new Error('safeClick failed after 3 attempts');
}

Pattern K — Tab Navigation

await page.getByRole('tab', { name: 'Related', exact: true }).click();
await expect(page.getByRole('tab', { name: 'Related' })).toHaveAttribute('aria-selected', 'true');

Pattern L — Field Error Validation

// Assert error appears after bad submit
const error = page.locator('.slds-form-element__help');
await error.waitFor({ state: 'visible', timeout: 5000 });
await expect(error).toContainText('Complete this field');

// Assert error clears after valid input
await error.waitFor({ state: 'hidden', timeout: 5000 });

Pattern M — Lightning Date Pickers

NEVER click the calendar icon or interact with the popup. Always fill the input directly in YYYY-MM-DD format.

// Preferred: label-based
const dateField = page.getByLabel('Close Date');
await dateField.waitFor({ state: 'visible', timeout: 30000 });
await dateField.fill('2026-12-31');
await dateField.press('Tab'); // trigger Salesforce validation

// API name approach
const dateField = page.locator('[data-field-api-name="CloseDate"] input');
await dateField.fill('2026-12-31');
await dateField.press('Tab');

// If value does not persist
await dateField.fill('2026-12-31');
await dateField.press('Tab');
await dateField.blur();

Applies to: Opportunity Close Date, Contract Start/End Dates, Quote Expiration Date, any standard Salesforce date field.

Pattern N — Related List / List View Row Actions

Target the row container first, open the action dropdown, then click the menu item — never click the record link text.

// Step 1: Locate and assert the row
const row = page.locator('[role="row"]').filter({ hasText: 'Acme Corp' }).first();
await row.waitFor({ state: 'visible', timeout: 30000 });
await expect(row).toContainText('Acme Corp');

// Step 2: Open action menu
const actionButton = row.locator('button[aria-haspopup="true"]').first();
await actionButton.click({ force: true });
await page.waitForTimeout(300); // allow menu render

// Step 3: Click scoped menu item
const menu = page.locator('[role="menu"]');
await menu.waitFor({ state: 'visible' });
await menu.locator('[role="menuitem"]').filter({ hasText: 'Edit' }).click();

Applies to: Related Lists (Quotes under Opportunity), List Views (Accounts, Opportunities), RLM Quote Line Items, any row-level action.

🔧 Self-Healing Deep Dive

Agent 6 (Self-Healer) runs automatically after every test execution. Here's how it classifies and fixes failures.

Failure TypeSymptomsFix Strategy
selector_failure Locator returned 0 or >1 elements, or timed out waiting for visibility Re-probe DOM via temp probe script, cross-reference scraped-locators.json, rewrite locator
timing_failure Action timed out while page was loading or spinner was active Add waitFor, increase timeout, inject waitForRlmSpinners() call
data_failure Lookup record doesn't exist, missing prerequisite data Add createSupportingRecord() helper in beforeAll
environment_failure Session expired, network error, org issue Not auto-fixable — written to healing-report.md with instructions

Healing limits and safety

The healer runs a maximum of 3 rounds per failure. After round 3, remaining failures are marked as unresolved and written to reports/healing-report.md with a manual action checklist.

The healer only patches code within // ── US-XXX START/END ── markers. Manually authored tests outside these markers are never touched.

Temporary probe files (probe-*.ts) used for DOM inspection are always deleted before the GitHub push step.
